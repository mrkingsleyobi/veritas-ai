# Documentation Contribution Guidelines

We welcome contributions to improve the Veritas AI documentation. This guide explains how to contribute effectively.

## How to Contribute

### Reporting Issues

If you find errors, unclear sections, or missing information:

1. Check if there's already an open issue for the problem
2. Create a new issue with:
   - Clear title describing the problem
   - Detailed description with examples
   - Suggested improvements (if any)
   - Screenshots or code examples (if relevant)

### Submitting Improvements

1. Fork the repository
2. Create a feature branch for your changes
3. Make your improvements
4. Test your changes locally
5. Submit a pull request with a clear description

## Documentation Standards

### Writing Style

- Use clear, concise language
- Avoid jargon and technical terms when possible
- Include practical examples
- Use active voice
- Maintain consistent terminology

### Formatting Guidelines

#### Headings
```markdown
# Main Title (H1)
## Section Title (H2)
### Subsection Title (H3)
#### Sub-subsection Title (H4)
```

#### Code Examples
````markdown
```python
# Python example
def verify_content(content_path):
    # Implementation here
    pass
```
````

#### Lists
```markdown
- First item
- Second item
- Third item

1. First step
2. Second step
3. Third step
```

#### Links
```markdown
[Link text](relative/path/to/document.md)
[External link](https://example.com)
```

### File Organization

- Place documentation in appropriate subdirectories
- Use descriptive filenames with hyphens (-) as separators
- Maintain consistent directory structure
- Update index/navigation files when adding new documents

## Content Guidelines

### Technical Accuracy

- Verify all code examples work correctly
- Test commands and configurations
- Ensure API endpoints and parameters are current
- Cross-reference related sections

### Completeness

Each document should include:
- Clear title and purpose
- Prerequisites (if any)
- Step-by-step instructions
- Expected outcomes
- Troubleshooting tips (if relevant)
- Next steps or related topics

### Examples and Use Cases

- Provide practical, real-world examples
- Include both basic and advanced use cases
- Show expected input and output
- Cover common error scenarios

## Review Process

### Before Submitting

- Proofread for spelling and grammar
- Verify all links work correctly
- Test code examples
- Ensure formatting is consistent
- Check mobile responsiveness (if applicable)

### Pull Request Requirements

- Clear, descriptive title
- Detailed description of changes
- Reference related issues (if any)
- Follow existing documentation style
- Pass all automated checks

## Style Guide

### Language and Tone

- Use American English
- Maintain professional but approachable tone
- Address readers as "you"
- Use present tense for instructions
- Use active voice instead of passive

### Technical Terms

- Use consistent terminology throughout
- Define acronyms on first use
- Link to glossary for complex terms
- Avoid internal project jargon

### Code Documentation

- Include comments in complex code examples
- Show both successful and error cases
- Provide context for code snippets
- Use realistic variable names
